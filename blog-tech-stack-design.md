## Problem Statement

为老刀AI码场（laodao-ai.cn）搭建一个基于 Markdown 的技术博客，作为内容中心和 SEO 引流渠道，与公众号/B站内容矩阵联动。核心挑战：国内访问速度优化 + 点赞 RSS 等功能集成 + 图片国内外双线存储。

## What Makes This Cool

- GitHub 存源码，push 触发自动构建，双线（国内阿里云 CDN + 国际 Cloudflare）同步发布
- 图片用 PicGo 一键上传阿里云 OSS，写 Markdown 和写普通文档无差别
- Waline 点赞 + 评论一体，需验证国内可达性（见风险项）
- 低月费运营（Railway Waline ~$5/月 + 阿里云 OSS 按量付费 + 域名年费，约 ¥50-100/月）

## Constraints

- Solo 开发者，不想维护服务器
- ICP 备案：已有备案主体和其它备案网站，预计审核周期较短（~1 周）；建站与备案**同步并行**，不互相等待
- 需要在国内访问速度有保障
- 点赞功能是必须的（作为读者反馈机制）
- 内容格式：Markdown，支持代码块 + 图片 + 标题导航

## Premises

1. 静态博客（SSG）是正确方向：零服务器维护，CDN 部署，SEO 友好
2. ICP 备案后可使用国内 CDN，这是国内速度的根本解法
3. 点赞必须依赖第三方服务（Waline），纯静态无法实现
4. 图片走 OSS + CDN，不存在 GitHub 仓库里（影响构建速度和 clone 效率）
5. 双域名策略（.cn 国内 + .com 国际）优于单域名，成本几乎为零

## Approaches Considered

### Approach A: Hugo + 阿里云 OSS/CDN + Waline（推荐）
- **核心**: Hugo SSG + GitHub 源码 + 阿里云 OSS/CDN（国内）+ Cloudflare Pages（国际）
- **Effort**: M（ICP 备案后约 1 周搭建完毕）
- **Risk**: 低
- **优点**:
  - Hugo 对 Go 开发者最友好，构建速度极快（毫秒级）
  - 阿里云 CDN 备案后国内速度最优
  - Waline 原生支持点赞，不需要单独实现
  - Hugo PaperMod/Stack 主题响应式，移动端开箱即用
- **缺点**:
  - ICP 备案需要等待时间（1-4 周）
  - Hugo Go template 语法学习成本（一次性）
- **Completeness**: 9/10

### Approach B: Astro + Cloudflare + 阿里云（现代双线）
- **核心**: Astro SSG + GitHub + Cloudflare Pages（国际）+ 阿里云 CDN（国内）
- **Effort**: M
- **Risk**: 中
- **优点**:
  - 组件化架构，添加交互元素更灵活
  - Astro 2026 Core Web Vitals 表现优异
- **缺点**:
  - Node.js 工具链，对 Go 开发者不如 Hugo 熟悉
  - 双线部署配置复杂度略高
- **Completeness**: 8/10

### Approach C: VitePress + GitHub Pages（极简快速上线）
- **核心**: VitePress + GitHub Pages + Cloudflare 代理
- **Effort**: S
- **Risk**: 中高
- **优点**: 零运维，最快上线
- **缺点**: 国内访问速度无保障，不适合长期 SEO 阵地
- **Completeness**: 6/10

## Recommended Approach

**Approach A — Hugo + 阿里云 OSS/CDN + Waline**

### 完整技术栈

| 模块 | 选型 | 说明 |
|------|------|------|
| SSG 框架 | Hugo | Go 生态，构建极快 |
| 源码托管 | GitHub (`github.com/laodao-ai`) | 博客源码在 `laodao-ai/blog`，内容系列代码各自独立 repo |
| 国内访问 | 阿里云 OSS + CDN | ICP 备案后，速度最优 |
| 国际访问 | Cloudflare Pages | 免费，全球 CDN |
| 图片存储 | 阿里云 OSS + CDN | 与博客静态文件同账号，PicGo 一键上传 |
| 点赞 + 评论 | Waline | 原生点赞，Railway 部署，LeanCloud 存数据 |
| RSS | Hugo 内置 | 开箱即用 |
| 分享按钮 | 自定义静态组件 | 微信/微博/知乎/X/LinkedIn |
| 移动端 | 主题内置响应式 | PaperMod 或 Stack 主题 |
| CI/CD | GitHub Actions | push 触发，双线同步发布 |

### 图片工作流

PicGo 工具 → 截图/拖拽 → 自动上传阿里云 OSS → 返回 CDN URL → 粘贴进 Markdown

### 点赞实现

Waline 文章级点赞：Railway 部署服务 + LeanCloud 国际版存数据，无需备案。

**点赞的定位**：反馈收集渠道（作者看后台数据），不是社交功能。点赞数对读者隐藏，按钮文案用"这篇对你有帮助吗？"而非"点赞"——语义更准确，反映的是实用价值而非喜好。

**评论功能：关闭。** 国内《互联网跟帖评论服务管理规定》（2022）要求实名制验证和内容审核，合规成本高且实名制本身会把评论量打到接近零。评论需求分流到其他渠道（见文章末尾设计）。

**⚠️ P0 风险 — 上线前必须验证：** Railway `*.railway.app` 域名在中国大陆存在 DNS 污染和封锁风险。点赞是反馈收集渠道，若国内加载失败则渠道直接失效，必须在上线前从中国大陆实际测试可达性。

Fallback 方案（按优先级）：
1. 给 Waline 服务绑定自定义域名（需要 Railway Pro，或迁移到其他平台）
2. 将 Waline 部署到阿里云函数计算（国内托管，速度最稳）
3. 将 Waline 部署到 Vercel + 自定义域名（Vercel 本身被封，自定义域名可绕过）

### RSS 订阅

Hugo 内置 RSS 生成，无需插件。构建后自动在 `/index.xml` 输出全站 feed，每个分类/标签也有独立 feed（`/categories/go/index.xml`）。

**配置要点（hugo.toml）：**
```toml
[outputFormats.RSS]
  mediaType = "application/rss+xml"
  baseName = "index"

[outputs]
  home = ["HTML", "RSS"]
  section = ["HTML", "RSS"]
```

**双域名策略下的 RSS：**
- 主订阅地址：`https://laodao-ai.cn/index.xml`（备案后）
- 备案前过渡期：`https://laodao-ai.com/index.xml`
- laodao-ai.com 的 RSS 中所有文章链接通过 `canonicalURL` 指向 .cn，避免 feed 聚合器显示重复条目
- 在博客 `<head>` 中放置 RSS autodiscovery 标签，让浏览器/RSS 阅读器自动发现：
  ```html
  <link rel="alternate" type="application/rss+xml" title="老刀AI码场" href="https://laodao-ai.cn/index.xml">
  ```

**订阅入口曝光：**
- 每篇文章末尾放 RSS 订阅引导（Hugo 模板统一注入）
- 首页 header 放 RSS 图标链接

### 分享按钮（纯静态）

静态博客没有后端，分享全部通过 URL scheme 或 JS 剪贴板实现，不依赖任何第三方 SDK。

**国内平台：**

| 平台 | 实现方式 | URL 模板 |
|------|----------|----------|
| 微信 | JS 复制链接到剪贴板 | 无 URL，提示用户"已复制，打开微信粘贴分享"（静态博客限制，已接受此 tradeoff） |
| 微博 | 跳转分享页 | `https://service.weibo.com/share/share.php?url={URL}&title={TITLE}` |
| 知乎 | 跳转分享页 | `https://www.zhihu.com/share?url={URL}` |

**国际平台：**

| 平台 | URL 模板 |
|------|----------|
| X / Twitter | `https://twitter.com/intent/tweet?url={URL}&text={TITLE}` |
| LinkedIn | `https://www.linkedin.com/sharing/share-offsite/?url={URL}` |

**Hugo 模板实现（在 layouts/partials/share.html 中统一管理）：**
```html
{{ $url := .Permalink }}
{{ $title := .Title }}
<div class="share-buttons">
  <button onclick="navigator.clipboard.writeText('{{ $url }}')">微信（复制链接）</button>
  <a href="https://service.weibo.com/share/share.php?url={{ $url }}&title={{ $title }}" target="_blank">微博</a>
  <a href="https://twitter.com/intent/tweet?url={{ $url }}&text={{ $title }}" target="_blank">X</a>
  <a href="https://www.linkedin.com/sharing/share-offsite/?url={{ $url }}" target="_blank">LinkedIn</a>
</div>
```

在每篇文章模板（`layouts/_default/single.html`）中 `{{ partial "share.html" . }}` 引入即可。

### 文章末尾反馈区设计

每篇文章末尾统一注入一个反馈区块（`layouts/partials/feedback.html`），整合点赞 + 评论分流：

```html
<div class="feedback-section">
  <p>这篇对你有帮助吗？</p>
  <!-- Waline 点赞组件，隐藏计数，仅显示按钮 -->
  <div id="waline-reaction"></div>

  <div class="discuss-links">
    <span>想进一步讨论？</span>
    <a href="{{ .Params.bilibili_url }}" target="_blank">B站视频评论区</a> ·
    <a href="https://github.com/laodao-ai/{{ .Params.github_repo }}/discussions" target="_blank">GitHub Discussions</a> ·
    <a href="/index.xml">公众号「老刀AI码场」</a>
  </div>
</div>
```

**评论分流逻辑：**

| 读者类型 | 评论去处 | 原因 |
|----------|----------|------|
| 国内读者 | B站视频评论区 | 已有账号，习惯这个动作 |
| 开发者读者 | GitHub Discussions | 技术讨论天然在这里；需要 GitHub 账号，自动筛选受众 |
| 私信反馈 | 公众号留言 | 关注后可留言 |

Front matter 中加 `bilibili_url`（对应视频链接）和 `github_repo`（对应系列 repo 名称）字段，无视频的文章可省略 B站链接。

### 关注引导

博客的关注引导需要区分国内外读者，因为平台完全不同。目标是把博客流量转化为各平台的长期关注者。

**国内读者：**

| 平台 | 引导方式 | 优先级 |
|------|----------|--------|
| 公众号 | 文章末尾放二维码 + "扫码关注，获取模板/资源" | ★★★（主要留存渠道） |
| B站 | 文章内嵌视频时放关注引导，或文末"视频版见B站" | ★★★ |
| 知乎 | 文末"同步发布于知乎，欢迎关注" + 知乎主页链接 | ★★ |
| 视频号 | 公众号文章内同步引导 | ★ |

**国际读者：**

| 平台 | 引导方式 | 优先级 |
|------|----------|--------|
| RSS | 页面 header + 文章末尾订阅按钮（最通用） | ★★★ |
| X / Twitter | 文末 "@laodaocheney on X" 链接 | ★★ |
| YouTube | 有英文视频内容时引导 | ★（目前占位，暂不主力） |

**Hugo 模板实现（layouts/partials/follow.html）：**

```html
<div class="follow-section">
  <!-- 国内：公众号二维码 -->
  <div class="follow-cn">
    <p>微信扫码关注「老刀AI码场」，获取 Spec 模板和工作流资源：</p>
    <img src="https://img.laodao-ai.cn/qrcode-wechat.png" alt="公众号二维码" width="120">
    <p>也可以 <a href="https://space.bilibili.com/668350177" target="_blank">关注「老刀AI码场」B站</a> 看视频版</p>
  </div>
  <!-- 国际：RSS + X -->
  <div class="follow-intl">
    <a href="/index.xml">Subscribe via RSS</a> ·
    <a href="https://x.com/laodaocheney" target="_blank">@laodaocheney on X</a>
  </div>
</div>
```

**区分国内外的方式：**

静态博客无法服务端判断 IP 归属，有两种做法：
1. **简单做法（推荐）**：两块引导区域同时展示，国内读者自然看公众号，国际读者自然看 RSS/X，不做区分
2. **复杂做法**：通过 JS 检测浏览器语言（`navigator.language`），中文语言环境优先展示公众号，其他语言优先展示 RSS——实现成本低，但覆盖不全面

初期用简单做法，内容量上来后再评估是否需要区分。

### 双域名策略

- laodao-ai.cn → 阿里云 CDN（国内，备案后生效）
- laodao-ai.com → Cloudflare Pages（国际，立即可用）

同一份静态文件，两个入口。**SEO 策略：选定 laodao-ai.cn 为主域名**，在 Hugo 配置中设 `canonicalURL = "https://laodao-ai.cn"`，Cloudflare Pages（.com）上的每个页面通过 `<link rel="canonical">` 指向 .cn 对应页面，避免双域名同内容导致 Google/百度 稀释权重。

### 代码块体验（开发者受众核心需求）

博客内容以 SDD 实战为主，每篇文章包含大量 Go 代码、YAML spec、终端命令，代码块体验直接影响读者是否愿意收藏和复用。

| 功能 | 优先级 | 实现方式 |
|------|--------|----------|
| 语法高亮 | P0 | Hugo 内置 Chroma，`pygmentsCodefences: true` |
| 一键复制按钮 | P0 | PaperMod 内置，`ShowCodeCopyButtons: true` |
| 语言标签显示 | P1 | PaperMod 自动从 fence 语言标识渲染 |
| 终端命令独立样式 | P2 | `bash`/`shell` fence + 自定义 CSS 深色背景 |
| 行号 | P2 | Hugo 支持，行号+复制按钮组合有已知样式 bug，上线前验证 |
| 长代码折叠 | P3 | 需自定义 JS，早期跳过 |

**必测场景**：Go 代码 + YAML + 终端命令混排时，移动端横向滚动体验。读者常在手机阅读文章后切回电脑操作，移动端可读性是 P1。

### GitHub ↔ 博客双向链路

博客文章和 GitHub 代码互为入口，两边都要显式互链，否则各自减半。

**博客文章 → GitHub（文章末尾固定区块）：**

```html
<!-- layouts/partials/github-link.html -->
<div class="github-link">
  <p>本文配套代码：
    <a href="{{ .Params.github_url }}" target="_blank">{{ .Params.github_label }}</a>
  </p>
</div>
```

Front matter 中加 `github_url` 和 `github_label` 字段，每篇文章各自指向对应 mode 目录。

**GitHub 账号结构（已确认）：**

```
laodaocheney (个人账号，管理员身份)
    └─► laodao-ai (Organization)
            ├── blog          ← 博客 Hugo 源码 + GitHub Actions
            ├── shorturl      ← 系列1: Go短链服务（含 mode1-4）
            └── [系列2]       ← 未来新系列各自独立 repo
```

**Repo 粒度原则：一个系列 = 一个 repo。** 系列内所有 mode 在同一 repo，Stars 集中，叙事完整，读者对比直观。

**GitHub README → 博客（系列配套表格）：**

```markdown
| 模式 | 内容 | 配套博客 |
|------|------|----------|
| mode1 | EP01 裸跑 | [laodao-ai.cn/posts/ep01-...](https://laodao-ai.cn/...) |
| mode4 | EP04 OpenSpec+SuperPowers | [laodao-ai.cn/posts/ep04-...](https://laodao-ai.cn/...) |
```

博客文章中 `github_url` 指向 `https://github.com/laodao-ai/shorturl/tree/main/shorturl-mode1`（具体 mode 子目录）。

## Open Questions

1. ~~Hugo 主题选择~~ **已决策：先用 PaperMod。** 早期内容少，Stack 的卡片布局和侧边栏反而显空旷；PaperMod 极简列表在这个阶段更合适，上线也更快。切换触发条件：**2个以上活跃系列 + 文章总数超过20篇**后重新评估 Stack。Hugo 换主题成本低（Markdown 不动，custom partials 基本可复用），不需要提前优化。
2. ~~ICP 备案主体~~ **已决策：企业主体备案。**
3. ~~是否需要评论功能，还是只要点赞？~~ **已决策：关闭评论，只保留点赞。** 国内监管要求实名制，合规成本高；评论需求分流至 B站/GitHub Discussions/公众号。
4. 是否同步公众号文章到博客（内容去重 SEO 策略）？公众号文章作为"原稿"，博客作为"结构化归档"，两者定位不同，可以同步，canonical 指向博客
5. RSS 主订阅地址：以 laodao-ai.cn/index.xml 为主，laodao-ai.com 的 RSS 在 Hugo 配置中禁用或重定向至 .cn

## Success Criteria

- RSS 可被 Feedly/Inoreader 正常订阅（主地址：laodao-ai.cn/index.xml）

## Distribution Plan

- 博客内容：`github.com/laodao-ai/blog` 存 Hugo 源码，GitHub Actions 构建，推送到阿里云 OSS + Cloudflare Pages
- 图片资源：阿里云 OSS + 阿里云 CDN（含国内节点），所有图片 URL 格式 `https://img.laodao-ai.cn/...`，统一走阿里云 CDN 分发（国内国际均通过阿里云 CDN，无需另接 Cloudflare）

## Next Steps

   ```yaml
   jobs:
     build:
       runs-on: ubuntu-latest

         - uses: actions/checkout@v4

       needs: build
       uses: cloudflare/pages-action  # 或 wrangler deploy
     deploy-aliyun:
       needs: build
       uses: manyuanrong/setup-ossutil@v2  # ossutil 推送到 OSS bucket
       # secrets: ALIYUN_ACCESS_KEY_ID, ALIYUN_ACCESS_KEY_SECRET, OSS_BUCKET
   ```
5. 申请 ICP 备案（阿里云备案入口 aliyun.com/icp），已有备案主体，步骤4-5可与步骤1-3**并行推进**。**备案期间 .cn 域名暂不开放，.com 作为主入口，同步积累 Google 权重**，备案完成后 .cn 接管国内流量，两边同时受益
6. 备案完成后：阿里云 OSS + CDN，.cn 域名切换到国内线，更新 Hugo canonicalURL
7. 配置阿里云 OSS + PicGo，迁移已有图片

## What I noticed about how you think

- 你主动说"域名已经改了，去读相关文件" — 你习惯把信息放在文件里而不是靠记忆，这是工程师思维，博客内容管理会很顺手。
- 你把博客定位成"内容中心 + SEO 引流"，不是"个人展示页" — 说明你清楚这个博客是整个内容矩阵的一个节点，不是终点。
- 一开始就提到国内访问速度和图片存储，不是上线后才想到的 — 你在设计系统，不是在拼积木。
